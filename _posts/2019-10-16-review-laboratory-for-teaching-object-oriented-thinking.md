# Introduction

초보자와 숙련된 절차적 프로그래머 모두에게 객체 지향 설계에 필요한 의인화 관점을 소개하는 것은 어렵습니다. 학습자가 객체에 대한 직접적인 경험을 할 수 있는 방법으로서 '클래스 이름'(classa name), '책임'(responsibilities), '협력자'(collaborators)로 객체를 특성화하는 CRC 카드를 소개합니다. 이러한 접근법은 초보 프로그래머에게 객체의 개념을 가르치고, 숙련된 프로그래머에게 복잡한 기존 디자인을 소개하는 데 성공했습니다.

# 1. Problem

객체 지향 프로그래밍을 가르치는 데에 있어 가장 어려운 문제는, 학습자가 절차적 프로그래밍으로 가능한 전체적인 지식(global knowledge)을 포기하고 객체가 가진 과제를 달성하기 위해 필요한 객체에 관한 로컬 지식(local knowledge of objects)에 의존하게 하는 것입니다. 초보적인 디자인은 전체적인 사고로 계속 회귀하게 됩니다. 즉 무의미한 전역 변수, 불필요한 포인터, 다른 객체의 구현에 대한 부적절한 의존으로 가득 차 있습니다.

객체에 대해 배우려면 위와 같이 전반적인 접근법을 변화시켜야 하기 때문에 객체를 가르치는 것은 객체의 디자인을 가르치는 것으로 바뀌게 됩니다. 우리는 초보자에게 기본 개념을 가르치거나 숙련된 객체 프로그래머에게 복잡한 디자인의 중요한 세부 사항을 가르치는 데 초점을 둡니다.

객체에 관해 자연스럽게 생각하는 학습 방법은, 객체 디자인을 절차적 디자인과 최대한 유사하게 만들려고 노력하는 게 아니라, 재료의 객체성에 몰입시키는 것(to immerse in the "object-ness" of the material)입니다. 이를 위해서 친숙한 자료를 가능한 많이 제거해야 합니다. 기본 사항을 철저히 이해한 후에 구문과 프로그래밍 환경 작동 같은 세부 사항을 빠르게 익히도록 기대하면서 말이죠.

이러한 맥락으로 우리는 객체 디자인, 구체적인 현상(concrete manifestation), CRC(클래스, 책임, 협력) 카드에 관한 우리의 관점(perspective)과 CRC 카드를 사용해서 객체로 생각(thinking with objects)하는 기본 사항(fundamentals)과 중요한 세부 사항(subtleties)을 가르쳤던 경험(experience)을 설명하겠습니다.

# 2. Perspective

절차적 디자인은 구현 언어나 운영 환경과 상관없이 프로세스, 데이터의 흐름, 데이터 저장을 갖는 추상적 수준으로 특징지을 수 있습니다. 우리는 객체 디자인에서도 유사한 기본 원칙을 제시하고 싶었습니다. 생각 끝에 객체의 역할을 식별하는 세 가지 차원(dimension)을 정했습니다. 클래스 이름, 책임, 그리고 협력자입니다.

객체의 클래스 이름은 디자인을 논의하는 동안 어휘 목록을 만들어냅니다. 많은 사람들이 말하길, 절차적 프로그램 디자인보다 객체 디자인이 언어 디자인과 공통점이 더 많습니다. 학습자는 (디자인하는 동안 충분한 시간을 들여서) 객체를 설명할 수 있는 올바른 단어 목록을 만들어야 합니다. 더 큰 디자인 환경의 맥락에서 내부적으로 일관되고 연상이 잘되는(evocative) 단어 목록말입니다.

책임은 해결해야 할 문제를 자세히 밝힙니다. 해결책은 다양한 버전과 개선점이 존재할 것입니다. 이때 책임은 잠재적 해결책을 결정하는 손잡이입니다. 객체의 책임은 능동형 동사(active verb)를 포함하는 짧은 동사구(verb phrases)로 표현할 수 있습니다. 이러한 방식으로 표현 가능할수록 디자인은 더 강력하고 간결해집니다. 다시 한 번 말하지만, 적절한 단어를 찾는데 사용하는 시간은 디자인에서 매우 가치 있는 시간입니다. 

객체 디자인에서 두드러지는 특징 중 하나는, 어떤 객체도 섬이 아니라는 점입니다. 모든 객체는 서비스 제공과 제어에 필요한 다른 객체와 관계 맺습니다. 마지막 차원은 객체의 협력자입니다. 책임을 수행하는 과정에서 필요한 메시지를 전달하거나 전달 받는 객체를 협력 객체라고 부릅니다. 협력이 꼭 대칭관계(symmetric relation)일 필요는 없습니다. 예를 들어, Smalltalk-80에서는 (아래 예제를 참고하세요) View와 Controller가 거의 동일하게 작동하지만 OrderedCollection은 클라이언트를 거의 또는 전혀 인식하지 못한 채로 서비스를 제공합니다.

본 글에서 우리는 의도적으로 클래스와 인스턴스를 엄밀하게 구별하지 않았습니다. 이러한 약식 행위는 생각만큼 혼란을 주지는 않습니다. 왜냐하면 우리가 제안하는 방법이 구체적이어서 인스턴스의 이름 짓기를 대체하기 때문입니다. 게다가 이렇게 함으로써 클래스 또는 프로토타입 기반 언어의 사용 여부와 무관하게 학습할 수 있습니다.
